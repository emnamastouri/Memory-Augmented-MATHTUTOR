RESUME COMPLET DU PROJET MATHTUTORAI
====================================

Nom du projet
-------------
MathTutorAI / Memory-Augmented-MATHTUTOR

Type de projet
--------------
Application web educative developpee avec Python et Streamlit.

Objectif general
----------------
MathTutorAI est une plateforme de tutorat mathematique augmentee par l'intelligence artificielle.
Elle cible les eleves du baccalaureat tunisien et permet de generer des exercices mathematiques,
de les valider automatiquement, de discuter avec un tuteur IA, de suivre la progression des eleves
et de permettre aux enseignants de gerer des groupes et des devoirs.

Contribution principale du projet
---------------------------------
La contribution principale n'est pas la simple reutilisation du dataset.
La contribution est une chaine complete :

1. generation d'exercices par LLM via OpenRouter ;
2. adaptation par memoire a partir d'exercices existants ;
3. alignement avec le programme officiel ;
4. validation par un juge LLM ;
5. validation locale deterministe avec SymPy et validateurs specialises ;
6. porte finale de presentation qui bloque tout exercice dangereux, incomplet ou mal forme ;
7. persistance MongoDB pour les utilisateurs, exercices, conversations et progression.

Pipeline de generation d'exercices
----------------------------------
Le pipeline suit ces etapes :

1. L'utilisateur choisit section, theme, sous-theme et type d'exercice.
2. Le systeme verifie que le couple section/theme/sous-theme existe dans le referentiel officiel.
3. Le module de memoire recupere des cas similaires dans le dataset.
4. Le prompt OpenRouter est construit avec :
   - consignes strictes ;
   - schema JSON ;
   - exemples de memoire ;
   - erreurs precedentes a eviter.
5. OpenRouter genere un exercice au format JSON.
6. Le JSON est extrait, repare si possible et valide contre un schema.
7. Un juge LLM verifie l'alignement et la coherence pedagogique.
8. Un validateur de solution verifie la coherence solution/reponse attendue.
9. Les validateurs deterministes locaux verifient les maths selon le domaine.
10. Le format LaTeX et l'affichage etudiant sont controles.
11. La porte finale decide si l'exercice peut etre affiche.
12. Si l'exercice est valide, il est stocke et presente.
13. Sinon, il est rejete, journalise et une nouvelle tentative peut etre lancee.

Schema structure d'un exercice
------------------------------
Un exercice genere doit contenir :

- title ;
- context ;
- questions ;
- instruction ;
- solution ;
- expected_answer ;
- answer_kind ;
- solution_steps ;
- learning_objective ;
- estimated_time ;
- table_data ;
- chart_data ;
- graph_data ;
- generation_metadata.

Regles importantes :
- le contexte seul est insuffisant ;
- il faut au moins deux questions explicites ;
- la solution doit repondre a toutes les questions ;
- la reponse attendue doit correspondre a la solution ;
- les donnees numeriques doivent etre coherentes ;
- aucun exercice mal forme ne doit etre affiche.

Memoire adaptative
------------------
Le systeme utilise une memoire a plusieurs niveaux :

1. Memoire dataset
   - recuperation d'exercices proches dans data/data-set.json ;
   - filtrage par section, theme et sous-theme ;
   - similarite textuelle ou TF-IDF selon les dependances disponibles.

2. Memoire positive
   - les generations LLM validees sont favorisees dans les prochaines tentatives.

3. Memoire negative
   - les echecs sont memorises pour eviter de repeter les memes erreurs :
     - JSON invalide ;
     - enonce sans question ;
     - probabilites incoherentes ;
     - LaTeX corrompu ;
     - reponse attendue contradictoire ;
     - exercice trop proche d'un cas source.

4. Anti-copie
   - le systeme bloque une generation trop similaire au cas source.

Porte finale d'affichage
------------------------
Un exercice n'est affiche que si :

- il vient d'une generation LLM reelle, sauf mode demo explicite ;
- le juge l'approuve ;
- l'alignement officiel est valide ;
- la solution est validee ;
- le validateur local ou domaine approuve ;
- le format LaTeX est propre ;
- les questions sont presentes ;
- chaque question est couverte par la solution ;
- les supports visuels requis existent ;
- aucune incoherence mathematique locale n'est detectee ;
- le dataset fallback n'est pas utilise silencieusement.

Le systeme est donc volontairement fail-closed :
si un doute important existe, l'exercice est bloque.

Protection du format LaTeX
--------------------------
Le projet repare ou bloque les corruptions frequentes :

- extbf{...} -> \textbf{...}
- extit{...} -> \textit{...}
- rac{...}{...} -> \frac{...}{...}
- hickapprox -> \approx
- extasciitilde -> \sim
- egin{cases} -> \begin{cases}
- end{cases} -> \end{cases}
- commandes trop echappees comme \\\\ln ou \\\\frac.

Les notations corrompues restantes sont bloquees avant affichage.

Mode dataset demo
-----------------
Le dataset peut servir de mode demonstration ou offline.
Il ne doit pas masquer un echec OpenRouter en mode normal.

En mode normal :
- si OpenRouter echoue ;
- si la generation est invalide ;
- si les validations echouent ;

alors l'application affiche une raison d'echec au lieu de montrer silencieusement un exercice fallback.
